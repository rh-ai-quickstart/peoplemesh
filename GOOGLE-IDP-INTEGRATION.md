# Google IDP Integration Strategy for Peoplemesh on OpenShift

## Overview

This document outlines the strategy for integrating your company's Google IDP with Peoplemesh deployed on OpenShift.

**Source Documentation**: [Peoplemesh OIDC Configuration Guide](https://github.com/francescopace/peoplemesh/blob/main/docs/how-to/configure-oidc.md)

## Architecture

```
User Browser
    ↓
OpenShift Route (https://peoplemesh.apps.your-cluster.com)
    ↓
Peoplemesh Service (port 8080)
    ↓
Peoplemesh App → /api/v1/auth/login/google
    ↓
Google OAuth Server (company Google Workspace)
    ↓
Callback → https://peoplemesh.apps.your-cluster.com/api/v1/auth/callback/google
    ↓
Peoplemesh creates session
    ↓
User redirected to app with authenticated session
```

## Key Components

### 1. OAuth Redirect URI

**Format**: `https://<peoplemesh-route-host>/api/v1/auth/callback/google`

**Example**: 
- If your OpenShift route is `peoplemesh-peoplemesh.apps.cluster-xyz.example.com`
- Then redirect URI is: `https://peoplemesh-peoplemesh.apps.cluster-xyz.example.com/api/v1/auth/callback/google`

**Critical**: This URI must be:
- HTTPS (OpenShift routes provide TLS termination automatically)
- Exact match in Google Cloud Console OAuth client configuration
- Accessible from user browsers (not internal cluster DNS)

### 2. Required Environment Variables

The Helm chart will configure these via Kubernetes Secret:

```yaml
OIDC_GOOGLE_CLIENT_ID: <from-google-console>
OIDC_GOOGLE_CLIENT_SECRET: <from-google-console>
SESSION_SECRET: <auto-generated-32-bytes>
OAUTH_STATE_SECRET: <auto-generated-32-bytes>
CORS_ORIGINS: "*"  # or specific domain for production
```

### 3. Google Cloud Console Setup

**Prerequisites**:
- Access to your company's Google Cloud Console
- Permissions to create OAuth 2.0 client IDs
- Google Workspace domain (e.g., `yourcompany.com`)

**Steps**:

1. **Navigate to Google Cloud Console**
   - Go to console.cloud.google.com
   - Select or create a project for peoplemesh

2. **Enable Required APIs** (may already be enabled)
   - Google+ API (for user profile information)
   - Google People API (optional, for richer profile data)

3. **Configure OAuth Consent Screen**
   - Go to: APIs & Services → OAuth consent screen
   - User Type: **Internal** (only for your Google Workspace organization)
   - App name: `Peoplemesh` or `<Company> People Search`
   - User support email: Your IT/support email
   - Authorized domains: Add `yourcompany.com`
   - Scopes: Default scopes are sufficient (email, profile, openid)

4. **Create OAuth 2.0 Client ID**
   - Go to: APIs & Services → Credentials
   - Click: Create Credentials → OAuth client ID
   - Application type: **Web application**
   - Name: `Peoplemesh - OpenShift Production`
   - Authorized redirect URIs:
     - Add: `https://<your-route>/api/v1/auth/callback/google`
     - Example: `https://peoplemesh-peoplemesh.apps.cluster-xyz.example.com/api/v1/auth/callback/google`
   - Click: Create
   - **Save the Client ID and Client Secret** (you'll need these for Helm)

## Helm Chart Configuration Strategy

### Current State

The Helm chart already has the structure in place:

```yaml
# charts/peoplemesh/templates/secrets.yaml
{{- if .Values.security.oidc.google.clientId }}
OIDC_GOOGLE_CLIENT_ID: {{ .Values.security.oidc.google.clientId | quote }}
OIDC_GOOGLE_CLIENT_SECRET: {{ .Values.security.oidc.google.clientSecret | quote }}
{{- end }}
```

### Integration Approach

**Option 1: Helm Values (Recommended for Testing)**

Pass credentials directly via `--set`:

```bash
helm install peoplemesh peoplemesh-umbrella/ \
  --set peoplemesh.security.oidc.google.clientId="123456789-abc.apps.googleusercontent.com" \
  --set peoplemesh.security.oidc.google.clientSecret="GOCSPX-xyz123..."
```

**Option 2: Values File (Recommended for Production)**

Create `values-production.yaml`:

```yaml
peoplemesh:
  security:
    corsOrigins: "https://peoplemesh.yourcompany.com"
    oidc:
      google:
        clientId: "123456789-abc.apps.googleusercontent.com"
        clientSecret: "GOCSPX-xyz123..."
```

Then install:

```bash
helm install peoplemesh peoplemesh-umbrella/ -f values-production.yaml
```

**Option 3: External Secret (Best for Production)**

If your organization uses External Secrets Operator or Vault:

1. Store credentials in your secret management system
2. Create Kubernetes Secret via External Secret
3. Modify Helm chart to use existing secret instead of creating one

### Route Hostname Determination

**Challenge**: The OAuth redirect URI needs the route hostname, but OpenShift auto-generates it.

**Solution Options**:

**A. Auto-Discovery (Current Implementation)**
- Let OpenShift auto-generate the route
- After deployment, retrieve it: `oc get route peoplemesh -n peoplemesh -o jsonpath='{.spec.host}'`
- Configure this URL in Google Console
- ✅ **Pros**: Simple, works immediately
- ❌ **Cons**: Two-step process (deploy, then configure Google)

**B. Pre-Define Route Hostname (Recommended)**
- Set a custom hostname in Helm values:
  ```yaml
  peoplemesh:
    route:
      host: "peoplemesh.yourcompany.com"
  ```
- Configure DNS CNAME: `peoplemesh.yourcompany.com` → `<openshift-router-domain>`
- Configure Google OAuth with known URL
- Deploy with predetermined URL
- ✅ **Pros**: Single deployment, predictable URL
- ❌ **Cons**: Requires DNS access

## Verification Strategy

### 1. Pre-Deployment Verification

```bash
# Verify route will be created
helm template peoplemesh peoplemesh-umbrella/ -f values-production.yaml | grep -A 10 "kind: Route"

# Verify secrets will contain OIDC variables
helm template peoplemesh peoplemesh-umbrella/ -f values-production.yaml | grep -A 20 "kind: Secret" | grep OIDC
```

### 2. Post-Deployment Verification

```bash
# Get the route URL
ROUTE_URL=$(oc get route peoplemesh -n peoplemesh -o jsonpath='{.spec.host}')
echo "Peoplemesh URL: https://$ROUTE_URL"

# Verify OIDC environment variables are set
oc exec deployment/peoplemesh -n peoplemesh -- env | grep OIDC

# Test the info endpoint (shows available auth providers)
curl -k https://$ROUTE_URL/api/v1/info | jq '.authProviders'

# Expected output:
# {
#   "loginProviders": ["google"],
#   "profileImportProviders": ["google"]
# }
```

### 3. End-to-End Login Test

1. Navigate to: `https://<route>/`
2. Click "Sign in with Google"
3. Should redirect to Google OAuth consent
4. After consent, should redirect back to peoplemesh
5. Verify authenticated: `curl -k https://<route>/api/v1/me` (with session cookie)

## Security Considerations

### 1. Internal User Type
- Use **Internal** OAuth consent screen type
- Restricts login to `@yourcompany.com` domain only
- Prevents external users from attempting authentication

### 2. Session Secrets
- `SESSION_SECRET` and `OAUTH_STATE_SECRET` are auto-generated (32 bytes)
- Stored in Kubernetes Secret (encrypted at rest)
- Used for session cookie signing and CSRF protection

### 3. CORS Configuration
For production, set specific origin instead of wildcard:

```yaml
security:
  corsOrigins: "https://peoplemesh.yourcompany.com"
```

### 4. HTTPS Enforcement
- OpenShift Route provides TLS termination
- Automatic redirect from HTTP to HTTPS
- No application-level TLS config needed

## Troubleshooting Guide

### Issue: "Provider google not found in /api/v1/info"

**Cause**: Environment variables not injected

**Solution**:
```bash
# Check secret exists
oc get secret peoplemesh-secrets -n peoplemesh

# Verify values
oc get secret peoplemesh-secrets -n peoplemesh -o jsonpath='{.data.OIDC_GOOGLE_CLIENT_ID}' | base64 -d

# Restart deployment to reload secrets
oc rollout restart deployment/peoplemesh -n peoplemesh
```

### Issue: "redirect_uri_mismatch" error from Google

**Cause**: Redirect URI in Google Console doesn't match actual callback URL

**Solution**:
1. Get actual route: `oc get route peoplemesh -n peoplemesh`
2. Verify callback URL: `https://<route-host>/api/v1/auth/callback/google`
3. Update Google Console OAuth client with exact URL
4. URL must match exactly (no trailing slash, correct protocol)

### Issue: Session not persisting after login

**Cause**: `SESSION_SECRET` missing or CORS misconfiguration

**Solution**:
```bash
# Verify SESSION_SECRET exists
oc exec deployment/peoplemesh -n peoplemesh -- env | grep SESSION_SECRET

# Check CORS_ORIGINS allows your domain
oc exec deployment/peoplemesh -n peoplemesh -- env | grep CORS_ORIGINS
```
